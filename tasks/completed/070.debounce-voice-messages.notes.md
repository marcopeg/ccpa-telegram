# Notes: 070 — Debounce voice messages (reverted)

**Status**: Reverted — task was based on a false assumption. All code changes undone.

---

## What we did

The task was created on the assumption that Telegram automatically splits long voice recordings (> 60 seconds) into multiple consecutive voice message updates, similar to how it splits long text messages at the 4096-character boundary. Based on this, we implemented:

- A per-user voice debounce buffer in `src/bot/handlers/voice.ts` (mirroring the text debounce in `text.ts`)
- A restructure of `config.debounce` from flat `{ windowMs }` to nested `{ text: { windowMs }, voice: { windowMs } }` in `src/config.ts`
- A `transcription.debugMsg` config flag to emit permanent debug messages showing audio parts received and the full transcript
- New docs: `docs/config/debounce/README.md`, updates to `docs/config/README.md` and `docs/config/reference.yaml`
- Wiring of a shared `debounceActiveUsers` set to the voice handler in `src/bot.ts`

Several iterations of the voice debounce logic were written to address a race condition (part 2 arriving while part 1 was still transcribing) before shifting to a "buffer-then-transcribe" approach to avoid concurrency entirely.

---

## What we found

**Telegram does not split voice messages.** Ever.

Research against the official Telegram Bot API documentation (`core.telegram.org/bots/api`) and community sources confirmed:

- Voice messages (`Voice` object) are always delivered as a single `Update` with a single `Message`
- There is no documented duration limit on voice messages at the Bot API level
- There is no media-group / split mechanism for voice (unlike photos/videos which can be sent as albums)
- The `duration` field is "as defined by the sender" with no enforced cap

The 60-second assumption was wrong. A 1m31s recording arrived as a single file — confirmed via `transcription.debugMsg: true` which reported "1 audio file received".

---

## The real constraint: 20 MB download limit

The actual limit that matters is file size, not duration:

- The hosted Telegram Bot API limits `getFile` downloads to **20 MB**
- Files over 20 MB cannot be downloaded via the standard Bot API — `getFile` returns an error
- To handle files > 20 MB, a **local Bot API server** is required
- At typical mobile Opus bitrates (~32 kbps), 20 MB ≈ 87 minutes of audio — so this limit is rarely hit in practice but is a real crash path

The current voice handler calls `getFile` with no size guard and no user-friendly error. This is tracked in task 075.

---

## Why reverted

The entire debounce-for-voice feature solved a problem that does not exist. All code, config schema changes, and documentation were reverted to the pre-task state.
