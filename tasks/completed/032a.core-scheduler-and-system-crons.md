## Summary

Introduce the core cron scheduler infrastructure and system-level cron support. Builds the scheduler engine, source file loaders (`.md` and `.mjs`), action executors, execution logging, and system-tier cron loading from `{configDir}/.hal/crons/`. No project-level or user-level support yet.

## Cron Source Format

System crons live in `{configDir}/.hal/crons/` as either `.md` (prompt-based) or `.mjs` (programmatic) files. There is no YAML-based format and no inline `crons` config key.

### `.md` — Prompt-based cron

The file is a Markdown document with YAML frontmatter. The body is the prompt sent to the target project engine.

```markdown
---
enabled: true
schedule: "0 9 * * *"           # cron expression (recurring) — OR —
# runAt: "2026-03-10T10:00:00Z" # ISO 8601 absolute datetime (one-off)

targets:
  - projectId: my-project       # required; which project engine to call
    userId: 123456              # optional; inject this user's context into the prompt
    flowResult: true            # optional; if true, send agent response to userId via Telegram
---

Check git status across all repositories and summarise pending changes.
```

**`enabled` defaults to `false`.** A cron file without `enabled: true` is loaded and validated but never scheduled. This makes it safe to commit draft crons to version control without them firing.

**`name` is always derived from the filename** (without extension). There is no `name` field in the frontmatter.

**Validation rules (enforced at boot; hard error):**
- `flowResult: true` requires `userId` — if `userId` is absent, this is a configuration error and the process must not start.
- During hot reload the same condition is treated as a logged error; the job is skipped but the process continues.
- `targets` must contain at least one entry; an empty array is a hard error.

**Execution model:**
- Before calling the engine, the prompt is wrapped with a `# Context` block (same format as user-driven messages) containing `sys.*`, `project.*`, `engine.*`, and `bot.userId` vars. This gives the engine project awareness without manual prompt engineering.
- The wrapped prompt is submitted as an anonymous call (`agent.call()`) to each target project's engine.
- `userId` (if set) is included in the context block as `bot.userId`. Full user-scoped session injection is deferred to 032b.
- `flowResult: true` (if set) causes the agent's response to be sent as a Telegram DM to `userId`.
- If `flowResult` is omitted or `false`, execution is silent (log-only).
- One log file is written **per target** (not per job). For multi-target crons, each target produces its own log file with the project ID in the filename.
- When a `projectId` is not found at runtime, a fallback log entry is written with `sys.*` context and an error status. Execution continues for remaining targets.

### `.mjs` — Programmatic cron

The file is an ES module that exports a scheduling declaration and a handler function.

```js
// Scheduling: export exactly one of `schedule` (recurring) or `runAt` (one-off).
export const enabled = true;
export const schedule = "0 9 * * *";
// export const runAt = "2026-03-10T10:00:00Z"; // ISO 8601 one-off

/**
 * @param {CronContext} ctx
 */
export async function handler(ctx) {
  // ctx.config                    — full computed HAL configuration in use
  // ctx.projects                  — map keyed by project slug
  // ctx.projects[key].config      — project config slice
  // ctx.projects[key].bot         — Grammy Bot instance (full Telegram API access)
  // ctx.projects[key].call(prompt) — call the project's AI engine; returns Promise<string>

  const project = ctx.projects["my-project"];
  const joke = await project.call("Tell me a Chuck Norris joke.");
  await project.bot.api.sendMessage(123456, joke);
}
```

**`enabled` defaults to `false`.** A `.mjs` file without `export const enabled = true` is loaded and validated but never scheduled.

**`name` is always derived from the filename** (without extension). There is no `name` export.

**Execution model:**
- The handler is called with a `CronContext` object (shape above).
- `ctx.projects[key].call(prompt)` invokes the project's AI engine anonymously and returns the response string.
- The handler is responsible for any output or side effects; there is no automatic result routing.
- Errors thrown by the handler are caught, logged, and do not kill the process.

## One-off Job Behaviour

A job with `runAt` set to a datetime in the past is **silently skipped** at load time (not scheduled). No error is raised; a debug-level log entry is written.

To re-run a past one-off:
1. Manually update `runAt` to a future datetime in the source file.
2. The file watcher detects the change and re-arms the job automatically.

There is no automatic "mark as done" mutation of source files.

## Scheduling

Croner supports both 5-field (minute precision) and 6-field (seconds precision) cron expressions. The optional first field is seconds, enabling sub-minute scheduling:

```
┌──────────────── second (0-59)      [optional]
│ ┌────────────── minute (0-59)
│ │ ┌──────────── hour (0-23)
│ │ │ ┌────────── day of month (1-31)
│ │ │ │ ┌──────── month (1-12)
│ │ │ │ │ ┌────── day of week (0-7)
* * * * * *   ← 6-field (with seconds)
  * * * * *   ← 5-field (standard)
```

Examples: `"*/10 * * * * *"` (every 10s), `"0 9 * * *"` (daily at 09:00).

## Scope

### In scope

- Add a scheduling library dependency (`croner`).
- Core scheduler engine: load cron source files, register timers, execute on schedule, manage lifecycle (start, stop, reschedule).
- Scheduling modes: recurring (`schedule`) and one-off absolute (`runAt`).
- Cron source loaders:
  - `.md` loader: parse YAML frontmatter + body, validate `targets`, apply validation rules.
  - `.mjs` loader: dynamic `import()` of the module, validate exports.
- Action executors:
  - **Prompt** (`.md`): full context injection via `buildCronContextVars()`, anonymous `agent.call()` per target, optional `flowResult` Telegram DM.
  - **Programmatic** (`.mjs`): invoke exported `handler(ctx)` with the full `CronContext`.
- File watcher on `{configDir}/.hal/crons/` for hot-reload:
  - File added → schedule new job.
  - File changed → replace existing scheduled job.
  - File deleted → remove scheduled job.
  - Validation errors on reload → log error, skip job, do not crash.
- Execution logging to `{configDir}/.hal/logs/crons/system/{job-name}.{md|mjs}/{timestamp}.{job-name}.txt`.
- `CronContext` construction: inject `ctx.config`, `ctx.projects[key].{config, bot, call}`.

### Out of scope (deferred to 032b/032c)

- Project-level crons (`{projectCwd}/.hal/crons/`).
- User-level crons (`{dataDir}/{userId}/crons/`).
- Slash commands (`/cron_add`, `/cron_list`, etc.).
- Relative one-off scheduling (`runIn`).
- `shell` action type.
- Full user-scoped context injection into `.md` cron agent calls (`userDir`, session).

## Requirements

- Add `croner` as a dependency.
- Implement a `CronScheduler` class that:
  - Accepts a `scope` string (e.g. `"system"`) prefixed to all log `jobId` fields as `{scope}/{name}`.
  - Accepts an array of loaded and validated cron definitions.
  - Registers timers for each definition whose trigger is in the future and `enabled === true`.
  - Executes the appropriate action handler when a timer fires.
  - Exposes methods to add, remove, and replace individual jobs at runtime (used by hot reload).
  - Provides a `stop()` method for clean shutdown.
- Implement a `.md` cron loader:
  - Parse frontmatter and body.
  - Validate required fields (`schedule` or `runAt`, at least one `targets` entry).
  - Enforce `flowResult` + `userId` co-presence rule (hard error at boot, logged error on hot reload).
  - Derive `name` from filename (no `name` frontmatter field).
  - `enabled` defaults to `false`.
- Implement a `.mjs` cron loader:
  - Dynamically import the module.
  - Validate presence of `schedule` or `runAt` and `handler` exports.
  - Derive `name` from filename (no `name` export).
  - `enabled` defaults to `false` (`mod.enabled === true` required to schedule).
- Implement a Zod schema for `.md` cron frontmatter.
- Implement `buildCronContextVars()` in `src/context/resolver.ts`:
  - Same pipeline as `resolveContext` but without a Grammy context.
  - Produces `sys.*`, `project.*`, `engine.*`, `bot.userId`, `bot.messageType: "cron"` vars.
  - Runs configContext merging, `${}` / `@{}` substitution, and both context hooks.
- Implement a `CronContext` builder:
  - Public `CronContext` includes `config`, `projects[key].{config, bot, call}`.
  - `call(prompt)` delegates to `createAgent(projectCtx).call(prompt)`.
  - Internal `internalProjectCtxs` map is kept separate (used by `.md` executor).
- Add a file watcher on `{configDir}/.hal/crons/` triggering add/change/delete handling.
- Wire the scheduler into the bot startup lifecycle: load all `.md` and `.mjs` files, start scheduler, stop on shutdown.
- Implement execution logging to disk:
  - Path: `{logBaseDir}/.hal/logs/crons/{scope}/{job-name}.{type}/{timestamp}.{job-name}.txt`
  - Scope subfolders: `system/` for system crons, `project/{slug}/` for 032b.
  - Extension in folder name (`.md` / `.mjs`) prevents collision between same-named crons.
  - For `.md` multi-target crons: one file per target, project ID appended to filename.
  - Log sections: header (job, source, project, started, ended, status), `--- prompt ---`, `--- context ---`, `--- project config ---`, `--- output ---`, `--- error ---`.
  - Fallback log entry for not-found projects (sys.* context, error status).

## Acceptance Criteria

- A `.md` cron fires on schedule and submits the prompt (wrapped with context) anonymously to the specified `projectId`.
- A `.md` cron with `userId` + `flowResult: true` sends the agent's response to that user via Telegram DM.
- A `.md` cron with `flowResult: true` but no `userId` causes a hard error at boot and a logged error (no crash) on hot reload.
- A `.mjs` cron fires on schedule and the exported `handler` is called with a correctly shaped `CronContext`.
- `ctx.projects[key].bot` provides a working Grammy Bot instance.
- `ctx.projects[key].call(prompt)` invokes the project's AI engine and returns the response string.
- Crons without `enabled: true` are loaded and validated but never scheduled.
- Recurring crons keep firing on schedule.
- Sub-minute scheduling works via 6-field cron expressions (e.g. `"*/10 * * * * *"`).
- A one-off cron (`runAt` in the future) fires exactly once and is not rescheduled.
- A one-off cron with a past `runAt` is silently skipped (not scheduled, debug log written).
- Adding a new file to `{configDir}/.hal/crons/` is detected and the job is scheduled without restart.
- Editing a file in `{configDir}/.hal/crons/` is detected and the previously scheduled job is replaced.
- Deleting a file from `{configDir}/.hal/crons/` is detected and the job is removed.
- A validation error in a hot-reloaded file is logged and the job is skipped; the process continues.
- Execution logs are written to `{configDir}/.hal/logs/crons/system/{job-name}.{type}/{timestamp}.{job-name}.txt`.
- For `.md` multi-target crons, one log file is written per target with the project ID in the filename.
- Log files include prompt body, injected context vars, and project config sections.
- Clean shutdown stops all scheduled timers.
- Invalid/malformed source files at boot cause a hard error; during hot reload they are skipped with a logged error.

## Implementation decisions (made during execution)

These decisions were taken outside the original scope during the implementation session.

### `enabled` defaults to `false`

Originally the plan had `enabled: z.boolean().default(true)`. Changed to `false` so cron files can be committed to version control without firing. Requires explicit `enabled: true` to schedule.

### `name` field removed

The plan had an optional `name` field in `.md` frontmatter and as a `.mjs` export. Removed entirely — the filename is always canonical. Both loaders derive `name` from `basename(filePath, ext)`. This eliminates a source of inconsistency where the frontmatter name and the filename could diverge.

### `scope` parameter on `CronScheduler`

The original plan's `CronScheduler` constructor had no `scope` parameter. Added `scope: string` so all log `jobId` fields are `{scope}/{name}` (e.g. `system/health-check`). This prepares 032b and 032c to pass their own scope strings (project slug, `"user"`) without changing the scheduler interface.

### Context injection for `.md` crons

The plan specified anonymous calls with `userDir: ""` and no context injection. During implementation, a new function `buildCronContextVars()` was added to `src/context/resolver.ts` that runs the same context-building pipeline as user-driven messages but without a Grammy context. The resulting vars (`sys.*`, `project.*`, `engine.*`, `bot.userId`, `bot.messageType: "cron"`) are prepended to the prompt body via `formatContextPrompt()`, giving the engine the same project awareness it has for user messages.

`buildSystemContext()` in `resolver.ts` was also exported to support the no-project fallback path.

### `ctx.projects[key].call(prompt)` on `CronProjectContext`

The original `CronProjectContext` only had `{ config, bot }`. A `call(prompt: string): Promise<string>` method was added so `.mjs` handlers can invoke the project's AI engine directly without going through internal APIs. Wired in `context.ts` as `createAgent(projectCtx).call(prompt)`. The internal `projectCtx` (with the full engine adapter) remains hidden from the public interface.

### Rich execution log format

The plan had a minimal log format: job, source, started, ended, status, output, error. Extended to include:
- `--- prompt ---` — raw prompt body before context injection (`.md` only)
- `--- context ---` — all resolved context vars key-by-key
- `--- project config ---` — slug, name, cwd, engine, model, session, context vars (no secrets)
- Per-target files for multi-target `.md` crons (project ID appended to filename)
- Fallback log entry when `projectId` is not found (previously a silent `continue`)

### Log directory structure

The plan had a flat `{logBaseDir}/.hal/logs/crons/{job-name}/` layout. Restructured to:

```
{logBaseDir}/.hal/logs/crons/
  system/
    {job-name}.md/   ← prompt-based crons
    {job-name}.mjs/  ← programmatic crons
  project/
    {slug}/          ← 032b
      {job-name}.md/
      {job-name}.mjs/
```

Reasons:
1. **Scope subfolder** (`system/`, `project/{slug}/`) separates log tiers — required for 032b.
2. **Extension in folder name** prevents collision between a `health-check.md` and `health-check.mjs` cron.

The `scope` and `type` fields were added to `CronLogEntry` and flow from `CronScheduler` → executor → `writeCronLog`.

### Sub-minute scheduling

Croner v10 supports 6-field cron expressions where the first field is seconds (`*/10 * * * * *` = every 10s). Confirmed and documented. Example `hello-world.mjs` updated to use `*/10 * * * * *`.

- Keep `CronScheduler` generic enough that 032b (project-level) and 032c (user-level) can reuse it.
