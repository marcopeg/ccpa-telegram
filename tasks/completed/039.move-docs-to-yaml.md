# 039 — Move docs to YAML

**Status:** completed

## Summary

Make YAML the single format for all config examples across the project. Add a **mega-config** under `docs/` that exposes and comments every possible config key, with comments linking to the relevant documentation pages — acting as a config-key–driven index for the docs. Add a **Configuration alternatives** section that describes JSON and JSONC with minimal examples only. Review and align README, `examples/`, `docs/`, and `.agents/skills` (including the docs skill) so documentation is consistent and the docs skill reflects how we write and keep docs up to date.

## Goals

1. **All examples in YAML** — Every config example in the repo (README, `examples/`, `docs/`, and any `.agents/skills` that show or reference config) uses YAML. No JSON/JSONC in primary examples.
2. **Mega-config as doc index** — Add the canonical full reference at **`docs/config/reference.yaml`** that:
   - Exposes and comments on **all possible config keys** (globals, engine, access, logging, rateLimit, context, commands, providers, projects, etc.).
   - In comments, **links to the relevant documentation pages** (e.g. Context, Commands, Engines, Rate limit, Logging) so the file doubles as an index into the docs from the config keys.
   - **`examples/hal.config.yaml`** is a shorter, copy-paste-friendly subset (not the full key list); the single source of truth for "every key" is `docs/config/reference.yaml`.
3. **Configuration alternatives section** — In the config docs, add a dedicated section (e.g. "Configuration alternatives" or "Other config formats") that:
   - States that `hal.config.json` and `hal.config.jsonc` are supported alongside `hal.config.yaml` (runtime support unchanged).
   - Documents which JSONC features are supported (`//`, `/* */`, trailing commas; no unquoted keys or single quotes).
   - Provides **one small example** for JSON and JSONC: some globals + one hypothetical project only (main config only; no `hal.config.local` example in this section). No full multi-project examples in those formats; users can convert from the YAML reference (e.g. with AI) if needed.
4. **Remove JSON/JSONC from examples/** — Remove any `hal.config.json` or `hal.config.jsonc` (or `_hal.config.jsonc`) from `examples/` so the folder contains only YAML example(s).
5. **Review and align** — Review README.md, `examples/`, and `docs/` so all config snippets and links use YAML and point to the right places. Review `.agents/skills` for references to example configs or doc structure (e.g. `refresh_models` references example config paths) and fix paths/format expectations. Ensure the **docs skill** (`.agents/skills/docs/SKILL.md`) is updated with current guidance: YAML-first for config examples, location and role of `docs/config/reference.yaml`, and how to write and keep documentation up to date. The docs skill must explicitly state: when adding or changing a config key, update `docs/config/reference.yaml` and the relevant doc page.

## Context and constraints

- **Config loader unchanged** — No changes to config loading or format detection; YAML and JSONC are already supported.
- **Example files** — `examples/` contains only YAML as the primary reference. No JSON/JSONC files in `examples/`; the Configuration alternatives section in the docs holds the minimal JSON/JSONC snippet (globals + one project) only.
- **Docs structure** — Prefer existing convention: one topic = one folder with `README.md`; other files in `docs/` use kebab-case. The mega-config is **`docs/config/reference.yaml`**.
- **Cross-links** — Use relative links in comments and docs so the documentation is navigable on GitHub.

## Out of scope

- Changing the config loader or adding new format support.
- Modifying any runtime behavior.
- **Init template** (`src/` init template / generated config format) — out of scope; full manual control stays with the user.

## Acceptance criteria

- [ ] Every config snippet in README.md, `docs/`, and any `.agents/skills` that show config uses YAML.
- [ ] **`docs/config/reference.yaml`** exists: all supported top-level and nested keys, with comments linking to the relevant doc pages. **`examples/hal.config.yaml`** is a shorter subset for copy-paste.
- [ ] Config docs include a "Configuration alternatives" (or equivalent) section: JSON and JSONC support explained, JSONC features listed, and one small example only (globals + one hypothetical project) for each format.
- [ ] `examples/` contains only YAML config example(s); any existing JSON/JSONC example files have been removed. Links from docs and README point to the YAML example(s).
- [ ] README, `examples/`, and `docs/` are reviewed for broken links, wrong format, or outdated paths; fixes applied.
- [ ] `.agents/skills` (e.g. refresh_models, docs) reference the correct example paths and format; docs skill describes YAML-first examples, `docs/config/reference.yaml` as the config-key reference, and explicitly says to update `reference.yaml` and the relevant doc page when adding or changing config keys.
