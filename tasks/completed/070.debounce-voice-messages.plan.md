### Step 3.2 — Extract `downloadAndTranscribe` helper

Create an internal async function inside `createVoiceHandler` (above the returned handler). This encapsulates the download+convert+transcribe steps so each part can be transcribed eagerly as it arrives.

- Takes `gramCtx`, `userId`, `voice`, and `messageId`.
- Returns `{ transcript: string; messageId: number }`.
- Downloads the OGA, converts to WAV via `convertToWav`, calls `transcribeAudio`, cleans up temp files.
- Does **not** send any status message.
- Throws on transcription failure (empty text) so the caller can surface the error.

### Step 3.5 — Add `flush` function

```ts
function flush(userId: number): void {
  const buf = buffers.get(userId);
  if (!buf) return;
  buffers.delete(userId);
  debounceActiveUsers.delete(userId);
  const sorted = [...buf.parts].sort((a, b) => a.messageId - b.messageId);
  const combined = sorted.map((p) => p.transcript).join(" ");
  dispatchTranscript(buf.gramCtx, combined).catch((err) => {
    logger.error({ err }, "Voice debounce flush error");
  });
}
```

Join separator is `" "` (space) rather than `""` used in text — voice transcripts are complete sentences and need a word boundary between them.

### Step 3.6 — Rewrite the outer returned handler

```ts
return async (gramCtx: Context): Promise<void> => {
  if (!config.transcription) {
    await gramCtx.reply(
      "Voice messages are not configured for this bot. Add a 'transcription' section to your project config.",
    );
    return;
  }

  const userId = gramCtx.from?.id;
  const voice = gramCtx.message?.voice;
  const messageId = gramCtx.message?.message_id;

  if (!userId || !voice) return;

  logger.info(
    { userId, duration: voice.duration, fileSize: voice.file_size },
    "Voice message received",
  );

  const existing = buffers.get(userId);

  if (voice.duration >= 60 || existing) {
    // Buffer this part: transcribe immediately, store, reset timer
    try {
      const part = await downloadAndTranscribe(gramCtx, userId, voice, messageId ?? 0);

      if (!existing) {
        // First part of a split — start buffer
        debounceActiveUsers.add(userId);
        buffers.set(userId, {
          parts: [part],
          timer: setTimeout(() => flush(userId), windowMs),
          gramCtx,
        });
      } else {
        // Subsequent part — reset timer
        clearTimeout(existing.timer);
        existing.parts.push(part);
        existing.gramCtx = gramCtx;
        existing.timer = setTimeout(() => flush(userId), windowMs);
      }
    } catch (error) {
      logger.error({ error }, "Voice transcription error during buffering");
      const errorMessage = error instanceof Error ? error.message : "Unknown error";
      await gramCtx.reply(`An error occurred processing the voice message: ${errorMessage}`);
    }
    return;
  }

  // Short message (duration < 60, no active buffer) — dispatch immediately
  try {
    const part = await downloadAndTranscribe(gramCtx, userId, voice, messageId ?? 0);
    await dispatchTranscript(gramCtx, part.transcript);
  } catch (error) {
    logger.error({ error }, "Voice handler error");
    const errorMessage = error instanceof Error ? error.message : "Unknown error";
    await gramCtx.reply(`An error occurred processing the voice message: ${errorMessage}`);
  }
};
```

```markdown
# Debounce

When a message or voice recording exceeds Telegram's size or duration limits, Telegram
automatically splits it into multiple consecutive updates. HAL buffers these parts per user
and sends the stitched content to the engine as a single request once the debounce window
expires with no new part arriving.

Debounce settings live under `globals.debounce` and can be overridden per project under
`projects.<key>.debounce`.

## Options

| Key | Description | Default |
|-----|-------------|---------|
| `text.windowMs` | Window in ms to collect consecutive Telegram-split text parts | `300` |
| `voice.windowMs` | Window in ms to collect consecutive Telegram-split voice parts | `4000` |

## Activation heuristic

Buffering is not unconditional — it activates only when a split is likely:

- **Text**: debouncing activates when the first received part is **≥ 4096 characters**
  (Telegram's single-message character limit). Shorter messages are dispatched immediately.
- **Voice**: debouncing activates when the first received voice message has
  **`duration ≥ 60` seconds** (Telegram's single-recording limit). Shorter recordings are
  dispatched immediately.

Once buffering is active for a user, any subsequent part from that user resets the timer.

## Voice buffering detail

Each voice chunk is **transcribed individually as it arrives**; transcripts are concatenated
in `message_id` order (ascending) with a space separator once the window expires. The combined
transcript is then sent to the engine as a single prompt. No status message is shown while
parts are being collected — the status appears only when the flush begins.

## Text buffering detail

Text parts are buffered as raw strings and concatenated (no separator) in `message_id` order
once the window expires. A `_Buffering..._` status message is shown while collecting parts.

## Why 4000ms for voice?

On a mobile connection, Telegram needs time to upload a ~60-second audio clip before the
next chunk is delivered to the bot. The 4000ms default accounts for that upload latency.
Reduce it if your users are on reliable high-speed connections; increase it if parts are
arriving after the window.
```

## Key decisions

| Decision | Rationale |
|----------|-----------|
| Join voice transcripts with `" "` (space) | Voice transcripts are complete sentences; no separator would run words together. Text parts use `""` because Telegram splits mid-word. |
| No status message while buffering voice | Task spec: no status during buffering. Users see nothing until the full transcript is being processed. |
| `downloadAndTranscribe` called eagerly (before flush) | Transcribe-then-join approach: each part is transcribed as it arrives, not after flush. Prevents chunk pile-up. |
| `debounceActiveUsers` shared set | Reuses the same set passed to `createRateLimitMiddleware`, which already skips counting for users in the set. No middleware changes needed. |
| Default 4000ms voice window | Accounts for mobile upload latency of a 60s clip. |
| No `statusMsgId` in voice `BufferEntry` | No status shown during buffering, so nothing to track or edit on flush. |
