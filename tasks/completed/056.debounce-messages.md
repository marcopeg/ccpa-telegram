# 056 — Debounce messages (text)

**Status**: draft

## Description

When a long text message (e.g. a pasted transcription) is sent to the bot, Telegram splits it into multiple consecutive updates of up to 4096 characters each. Currently each part is processed independently — the engine receives and responds to each fragment separately. HAL should detect when a message has been split by Telegram, buffer the consecutive parts, stitch them back together in the correct order, and send the full combined prompt to the engine as one request.

## Context & Constraints

- **Scope**: text messages only (`message:text` handler). Voice, photo, and document handlers are out of scope for this task.
- **Trigger**: Telegram splits long pasted text at exactly 4096 characters. Each part is delivered as a separate message update in rapid succession.
- **Buffering granularity**: per-user, per-bot. Two users sending messages simultaneously must not interfere with each other.
- **No existing buffering**: messages are currently processed immediately on arrival with no queue.
- **Bot library**: Grammy v1.39.2.
- **Relevant files**: `src/bot/handlers/text.ts`, `src/config.ts`, `src/bot/middleware/rateLimit.ts` (existing per-user in-memory state pattern to follow).

## Explicit Expectations

### Trigger condition
Debouncing is activated **only** when an incoming message has `message.text.length >= 4096`. Messages shorter than 4096 chars are considered standalone and processed immediately with no delay — no buffer, no timer.

When a 4096-char message arrives:
1. Send a **"Buffering…"** status message to the user as acknowledgement.
2. Start the debounce timer (`debounce.windowMs`, default `300ms`).
3. Add the message to the per-user buffer (keyed by Telegram user ID).

When subsequent messages arrive while the buffer is active (timer still running):
- Add the new part to the buffer regardless of its length (the last chunk of a Telegram split is always shorter than 4096).
- Reset the debounce timer.

When the timer fires (no new message within the window):
- Sort buffered parts by `message_id` (ascending) to guarantee correct order.
- Concatenate all parts with no separator (plain string join).
- Replace the "Buffering…" status message with "Processing…".
- Forward the combined prompt to the engine.
- Clear the buffer.

### Rate limiting
Rate limiting is applied once to the flushed (combined) message, not to each individual fragment.

### Slash commands
Slash commands are never buffered. A message starting with `/` is always dispatched immediately, even if it is 4096 chars long (edge case, but should not be debounced).

### Debounce window
Default `300ms`. Rationale: Telegram delivers split parts within ~50ms under normal conditions; 300ms provides a safe margin while keeping normal message latency at zero (since short messages bypass debounce entirely). Configurable via `debounce.windowMs` in project or global config (same pattern as `rateLimit.windowMs`).

## Acceptance Criteria

- [ ] Messages shorter than 4096 chars are processed immediately with no debounce delay.
- [ ] A message of exactly 4096 chars triggers the debounce: a "Buffering…" status is shown and subsequent parts are collected.
- [ ] A long pasted message split by Telegram into N parts is received by the engine as a single concatenated prompt.
- [ ] Parts are concatenated in `message_id` ascending order with no separator.
- [ ] "Buffering…" status is shown while collecting parts; replaced by "Processing…" when flushed.
- [ ] Debounce window defaults to 300ms and is overridable via `debounce.windowMs`.
- [ ] Buffer is per-user (keyed by Telegram user ID); simultaneous messages from different users do not interfere.
- [ ] Rate limiting is applied once per flushed message, not per individual fragment.
- [ ] Slash commands bypass debounce and are dispatched immediately.
- [ ] Existing text handler behaviour (engine execution, streaming status, slash commands) is unchanged for processed messages.
