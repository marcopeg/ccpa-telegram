# Plan: 056 — Debounce text messages

## Context

Telegram splits text messages longer than 4096 characters into multiple consecutive updates. Currently each fragment is dispatched to the engine independently, so a pasted transcription gets N separate engine calls with N separate responses. The fix is to detect when a 4096-char message arrives (the split trigger), buffer subsequent parts, and flush them as one combined prompt after a configurable quiet window.

```ts
  .object({ windowMs: z.number().positive() })
  .partial()
  .optional(),
```

**1b. Same addition to `ProjectFileSchema`.**

**1c. Add to the compiled config type:**
```ts
debounce: { windowMs: number };
```

**1d. Add default resolution (alongside `rateLimit` block):**
```ts
debounce: {
  windowMs: project.debounce?.windowMs ?? globals.debounce?.windowMs ?? 300,
},
```

## Phase 2 — Rate limit middleware (`src/bot/middleware/rateLimit.ts`)

Update `createRateLimitMiddleware` to accept an optional second parameter:

```ts
export function createRateLimitMiddleware(
  ctx: ProjectContext,
  debounceActiveUsers?: Set<number>,
)
```

Inside the middleware, before the existing rate-limit logic, skip counting for buffered parts (2..N):
```ts
if (userId && debounceActiveUsers?.has(userId)) {
  await next();
  return;
}
```

## Phase 3 — Text handler (`src/bot/handlers/text.ts`)

**3a.** Update function signature to accept `debounceActiveUsers: Set<number>`.

**3b.** Add debounce state inside the outer closure:
- `windowMs` from config
- `BufferEntry` interface (parts array, timer, gramCtx, statusMsgId)
- `buffers: Map<number, BufferEntry>`

**3c.** Extract engine-dispatch logic into inner `dispatchMessage(gramCtx, text, existingStatusMsgId?)`:
- If `existingStatusMsgId` provided → edit to `_Processing..._`
- Otherwise → send new `_Processing..._` message

**3d.** Add `flush(userId)` helper:
- Sort parts by `messageId` ascending
- Concatenate with no separator
- Call `dispatchMessage` with buffered `gramCtx` and `statusMsgId`

**3e.** Replace direct engine-dispatch block with debounce logic:
- `messageText.length >= 4096` or existing buffer → buffer/reset timer
- Otherwise → `dispatchMessage` immediately

## Key behaviours

| Scenario | Result |
|----------|--------|
| Short message (< 4096) | Dispatched immediately, no delay |
| 4096-char first part | "Buffering…" shown; buffer created; user added to `debounceActiveUsers` |
| Subsequent parts | Timer reset; parts appended |
| Timer fires | Parts sorted by `message_id`, concatenated, "Buffering…" → "Processing…", engine called once |
| Slash command | Bypasses debounce entirely |
| Rate limit (parts 2..N) | Skipped — user is in `debounceActiveUsers` |
| Rate limit (first part) | Counted normally |
| Concurrent users | Independent buffers per user |
