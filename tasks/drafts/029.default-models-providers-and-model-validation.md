# 029 — Default models: providers config and /model validation behavior

## Status

draft

## Summary

Add support for top-level `providers` config and a basic hardcoded list of available models per engine, then align `/model {name}` validation behavior when using potentially stale hardcoded data.

## Requirements

- Handle the top-level `providers` configuration in the app config flow.
- Add a basic hardcoded list of model IDs available for each engine.
- Use the hardcoded lists as fallback model metadata when dynamic sources are unavailable.
- Ensure `/model {name}` skips strict validation when the source of truth is the hardcoded list, because those lists may be outdated.
- Keep strict validation when model data comes from fresh/dynamic sources.

## Acceptance Criteria

- Config parser accepts and exposes top-level `providers` values.
- Each supported engine has an initial hardcoded model list.
- `/model {name}` does not block model selection solely due to hardcoded-list mismatches.
- Existing model-selection behavior remains unchanged when dynamic model lists are available.
