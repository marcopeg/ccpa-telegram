# 032b — Project-level crons + logging refinement

## Status

draft

> **Part of [032 — Cron jobs and scheduled actions](./032.cron-jobs-and-scheduled-actions.md).**
> Read the parent task first for shared specifications (YAML format, session behaviour, logging format, source tiers, lifecycle rules). This file only covers what is specific to 032b.

## Summary

Extend the cron scheduler to support project-level cron definitions (inline in project config blocks and file-based in `{projectCwd}/.hal/crons/`), add `runAs` session support, and refine execution logging based on experience from 032a.

## Dependencies

- [032a — Core scheduler + system-level crons](./032a.core-scheduler-and-system-crons.md) must be completed first.

## Scope

### In scope

- Project-level cron sources:
  - **Inline:** new `crons` key inside each `projects[n]` config block, validated by Zod.
  - **File-based:** `{projectCwd}/.hal/crons/*.yaml`, loaded at boot.
- Per-project scheduler instances: each project runs its own scheduler, managing both system-level and project-level crons relevant to that project.
- File watcher on `{projectCwd}/.hal/crons/` for hot-reload of project cron files.
- Config hot-reload: reconcile project inline crons on config change.
- `runAs: <userId>` session mode for project and system crons (as defined in 032).
- Refine execution logging format and contents based on learnings from 032a.
- Ensure system crons and project crons coexist without name collisions (namespace or prefix strategy).

### Out of scope (deferred to 032c)

- User-level crons (`{dataDir}/{userId}/crons/`).
- Slash commands (`/cron_add`, `/cron_list`, etc.).
- `user` session mode (only relevant for user-defined crons).

## Requirements

- Extend the config Zod schema to accept a `crons` array inside each `projects[n]` block.
- On project boot, merge system-level and project-level crons into the project's scheduler instance.
- Handle cron name uniqueness: project crons override system crons with the same name (or error — decide during planning).
- Add file watcher on `{projectCwd}/.hal/crons/` per active project.
- Implement `runAs` session resolution: given a userId, resolve the user directory, load/create session, and execute via the engine with that user's context.
- Refine log format based on 032a feedback (structure, readability, metadata).
- Ensure clean teardown per project (stop project scheduler when project is deactivated or config reloads).

## Acceptance Criteria

- A cron defined inline in a `projects[n].crons` block fires on schedule for that project.
- A cron defined as a YAML file in `{projectCwd}/.hal/crons/` fires on schedule.
- System crons and project crons both run within the same project scheduler without conflict.
- Project cron files are hot-reloaded (add/edit/delete detected and rescheduled).
- Config hot-reload picks up changes to project inline `crons`.
- `runAs` session mode correctly executes in the specified user's context.
- Deactivating a project (or removing it from config) stops its scheduler and cleans up watchers.
- Logging improvements from 032a feedback are applied.

## Notes

- The name collision strategy (override vs error) should be decided during task planning.
- Logging refinements are intentionally vague here — they depend on what we learn from running 032a in practice.
