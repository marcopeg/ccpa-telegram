# 070 — Debounce voice messages

**Status**: draft

## Description

When a voice recording exceeds Telegram's ~60-second limit, Telegram automatically splits it into multiple consecutive voice messages. Currently each part is processed independently by the voice handler — the engine receives and responds to each audio fragment separately. HAL should buffer consecutive voice messages from the same user, collect all parts within a debounce window, and send the full stitched prompt to the engine as one request.

## Context & Constraints

- **Scope**: voice messages only (`message:voice` handler). Related but distinct from task 068 (which stitches Whisper partial transcript output within a single voice message).
- **Trigger**: Telegram splits long voice recordings at ~60-second boundaries, delivering each chunk as a separate voice message update in rapid succession.
- **Relevant file**: `src/bot/handlers/voice.ts`, `src/config.ts`.
- **Debounce config key**: `debounce.voiceWindowMs` (separate from `debounce.windowMs` used for text, since the inter-message gap for voice splits may be larger).
- **Bot library**: Grammy v1.39.2.

## Open Questions

- **How to stitch audio?** Two approaches:
  1. **Transcribe-then-join**: transcribe each voice part independently as it arrives (in buffer order), then concatenate the transcriptions into a single prompt before sending to the engine. Simpler; relies on per-chunk transcription accuracy.
  2. **Download-then-join**: download all audio files, concatenate the raw audio, and transcribe once. Better accuracy at boundaries; more complex (requires audio manipulation).
  Which approach is preferred?

- **Debounce window default**: Voice splits arrive slightly slower than text splits (each chunk is a full audio upload). A larger default (e.g. 2–5 seconds) may be needed to account for upload time of each part. What's a reasonable default?

- **Slash commands in voice**: If the first voice part transcribes to a slash command, should the debounce still apply (collecting subsequent parts) or should commands be dispatched immediately?

## Acceptance Criteria (draft — pending open questions)

- [ ] Multiple consecutive voice messages (Telegram-split recording) are treated as a single prompt by the engine.
- [ ] Parts are buffered per-user and flushed once the debounce window expires with no new voice message.
- [ ] Debounce window is configurable via `debounce.voiceWindowMs` with a sensible default.
- [ ] Buffer is per-user; simultaneous voice messages from different users do not interfere.
- [ ] No "Processing…" status message is shown while buffering parts.
- [ ] A single (non-split) voice message is processed correctly after the debounce window.
- [ ] Existing voice handler behaviour (transcription, streaming status, engine execution) is unchanged for the flushed combined prompt.
