# 065 — Granular project hot reload

## Status

draft

## Summary

Replace the current full-restart hot reload strategy (all bots + schedulers stop and restart on any config change) with a granular reload that only stops and restarts the projects actually affected by a change. Projects unaffected by a config edit continue running without interruption.

## Context

The current hot reload cycle in `cli.ts` is all-or-nothing: any change to the main config file triggers `runBotsForConfig()` to tear down every bot, scheduler, and watcher, then restart everything. This is safe but disruptive — a change to one project config briefly interrupts all other projects.

As the system grows to support more per-project artefacts (project crons from 032b, project commands watcher, future user-level crons), the blast radius of a full reload grows. Granular reload becomes increasingly valuable.

## Scope

### In scope

- Diff the previous and next config to determine which projects were added, removed, or changed.
- For **added** projects: start their bot, project cron scheduler, command watcher, and any other per-project services.
- For **removed** projects: stop their bot, project cron scheduler, command watcher, and clean up all handles.
- For **changed** projects: stop then restart only that project's services.
- For **unchanged** projects: do nothing — they keep running.
- System crons (from `startSystemCrons()`) are reloaded separately only if the system-level config (`configDir/.hal/crons/` or global config keys) changed.
- A project handle (`ProjectHandle`) encapsulates all stoppable services for a single project: `{ bot, cronHandle, commandWatcher, stop() }`.

### Out of scope

- User-level crons (032c) — but the `ProjectHandle` shape should be extensible to include them later.
- Changes to the Telegram bot token requiring a full restart — these always require a full reload (token is set at bot construction time).
- Partial config validity (one project invalid, others fine) — decide during planning whether to skip invalid projects or abort.

## Requirements

1. Implement a config differ: compare previous and next resolved configs and return `{ added, removed, changed, unchanged }` project sets.
2. Define a `ProjectHandle` type that wraps all per-project stoppable services.
3. Refactor `runBotsForConfig()` into a `ProjectHandle`-based lifecycle where each project is independently startable and stoppable.
4. In the config watcher reload callback: compute the diff, stop removed/changed projects, start added/changed projects, leave unchanged projects running.
5. Handle the case where the Telegram bot token changed for a project: treat as a full remove + add (bot must be reconstructed).
6. System crons: reload only if global config or `configDir/.hal/crons/` changed (not on every project-level change).
7. Ensure the state map (`slug → ProjectHandle`) is kept consistent across reload cycles.

## Acceptance Criteria

- Editing one project's config in the main config file does not interrupt other projects' bots or schedulers.
- A project added to config starts its bot, cron scheduler, and command watcher without a full restart.
- A project removed from config stops its bot, cron scheduler, and command watcher cleanly.
- A project with an unchanged config slice is not touched during a reload.
- If a project's bot token changes, its bot is fully reconstructed.
- System crons are not reloaded when only a project-level config field changes.
- All per-project services (bot, cron scheduler, file watcher) are represented in `ProjectHandle` and stopped via a single `handle.stop()` call.

## Notes

- This task depends on 032b being complete, since project cron handles need to be included in `ProjectHandle`.
- The config differ needs to handle both structural changes (project added/removed) and value changes (field edited). A deep-equality check per project slug is sufficient.
- Bot token changes are detectable in the differ — flag them explicitly so the reload path knows to reconstruct the Grammy `Bot` instance.
- Consider whether `startSystemCrons()` should also return a way to reload individual system crons (rather than stop/restart all) — but that is a separate concern and should not block this task.
