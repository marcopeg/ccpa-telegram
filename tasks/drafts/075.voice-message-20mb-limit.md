# 075 — Handle voice messages exceeding Telegram's 20 MB download limit

**Status**: draft

## Problem

The Telegram Bot API limits file downloads via `getFile` to **20 MB**. If a user sends a voice message whose file size exceeds this threshold, `getFile` fails with a Telegram API error. The current voice handler has no guard for this case — the error falls through to a generic catch block and the user receives an unhelpful message: _"An error occurred processing the voice message: ..."_.

## Context & Constraints

- **Limit**: Standard hosted Bot API — `getFile` returns an error for files > 20 MB. The file is still referenced in the update (the `file_id` is present and `file_size` may be set), but it cannot be downloaded.
- **Practical threshold**: At typical Telegram mobile Opus bitrate (~32 kbps), 20 MB ≈ ~87 minutes of audio. Hitting this limit requires a very long recording, but it is a real crash path.
- **`file_size` is optional**: The `Voice` object's `file_size` field is `number | undefined` — Telegram does not always include it. A pre-flight size check is possible when the field is present but cannot be relied on in all cases.
- **Local Bot API server**: The 20 MB cap is specific to the hosted API. A local Bot API server raises it to 2 GB. HAL does not currently support local server configuration, so this is out of scope for this task.
- **No voice splitting**: Telegram does NOT split voice messages at any duration boundary. Each recording arrives as a single update regardless of length (confirmed by research — see `070.debounce-voice-messages.notes.md`).

## Current user experience

1. User sends a long voice message (> 20 MB file size)
2. Bot calls `getFile` → Telegram API returns an error
3. Catch block fires: user sees _"An error occurred processing the voice message: \<raw API error text\>"_
4. The status message may or may not have been sent before the error; cleanup may not run

The raw API error text is not user-friendly and gives no guidance on what to do.

## What to do

### Pre-flight check (when `file_size` is available)

Before calling `getFile`, check `voice.file_size` against the 20 MB threshold. If it exceeds the limit, reply with a clear, actionable message and return early — skip the download entirely.

```
Voice message is too large to process (file size: X MB).
The Telegram Bot API only supports voice files up to 20 MB (~87 minutes at standard quality).
```

### Graceful error on `getFile` failure (when `file_size` is absent)

Wrap the `getFile` call to detect the specific "file too large" error from Telegram (error code 400, description contains "file is too big") and surface a friendly message instead of the raw API error.

### Status message lifecycle

Ensure no orphaned status message is left in the chat if the error occurs before or after the status message is sent.

## Acceptance criteria

- [ ] If `voice.file_size` is present and > 20 MB: reply with a clear user-facing message, no download attempted, no error logged as an error (log as warn with file size).
- [ ] If `getFile` fails due to file size (Telegram error 400 "file is too big"): reply with the same user-facing message.
- [ ] No orphaned status message left in chat on either error path.
- [ ] All other `getFile` failures continue to surface as generic errors (no regression).
- [ ] Threshold constant is named and documented (not a magic number).
