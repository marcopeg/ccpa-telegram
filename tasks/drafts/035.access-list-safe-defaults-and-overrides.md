# 035 — Access list safe defaults and overrides

## Status

draft

## Summary

Harden access control defaults so the bot is not open to everyone unless explicitly and dangerously configured.

## Requirements

- Default behavior must be restrictive, not open.
- Access is allowed only when at least one of these is true:
  - `access.allowedUserIds` contains at least one user ID, or
  - `projects.<name>.access.allowedUserIds` contains at least one user ID, or
  - `access.dangerouslyAllowUnrestrictedAccess` is explicitly set to `true`, or
  - `projects.<name>.access.dangerouslyAllowUnrestrictedAccess` is explicitly set to `true`.
- Introduce/configure the explicit dangerous flag name: `dangerouslyAllowUnrestrictedAccess`.
- For the `access` key specifically, project-level config must be a full replacement of global `access` (not a merge/extension).
- Precedence rule:
  - if project defines `access`, use only project `access`;
  - otherwise use global `access`.
- Within an effective access object, `allowedUserIds` takes precedence over `dangerouslyAllowUnrestrictedAccess`.
  - If `allowedUserIds` is present/non-empty, only listed users are allowed.

## Acceptance Criteria

- If no effective `access` permits users and no dangerous flag is true, bot denies access.
- Global `dangerouslyAllowUnrestrictedAccess: true` no longer applies when a project defines its own `access`.
- Project `access` fully replaces global `access`.
- Non-empty `allowedUserIds` is enforced even when dangerous flag is true in the same effective scope.
- Misconfiguration is surfaced with clear logs/errors so unsafe defaults are visible.
