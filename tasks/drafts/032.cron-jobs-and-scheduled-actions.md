# 032 — Cron jobs and scheduled actions

## Status

## Summary

Add cron-job support to schedule tasks in time, including one-off timers and recurring schedules, with persistent storage so jobs survive hot reload and process restarts. Jobs can be defined at three levels (system, project, user), execute prompts or shell commands, and log every execution to disk.

## Shared Specifications

The sections below are the single source of truth for cross-cutting concerns. Sub-tasks reference this file rather than duplicating these definitions.

### Cron Definition Format

Each cron job is a single YAML file:

```yaml
name: daily-git-status
enabled: true

# --- Scheduling (exactly one of these) ---
schedule: "0 9 * * *"          # cron expression (recurring)
# runAt: "2026-03-05T10:00:00Z" # ISO 8601 datetime (one-off)
# runIn: "30m"                   # relative duration (one-off)

# --- Action (exactly one of these) ---
action:
  type: prompt                  # "prompt" | "shell"
  value: "Check git status and summarize pending changes"

# --- Session behaviour ---
session: user      # see Session Behaviour below

# --- Output targets (optional) ---
userIds: [123456]  # Telegram user IDs to receive output; omit for silent
```

**Scheduling:** exactly one of `schedule` (cron expression, recurring), `runAt` (ISO 8601, one-off), or `runIn` (duration like `30m`, one-off relative to load time). One-off jobs are marked `enabled: false` after execution.

**Action types:** `prompt` (sent to the project engine) or `shell` (subprocess in project `cwd`). Skills can be invoked via prompt using `/skill_name` syntax.

### Session Behaviour

| Cron tier         | Default     | Options                                              |
|-------------------|-------------|------------------------------------------------------|
| User-defined      | `user`      | `user` (owning user's session) · `anonymous` (fresh) |
| Project / System  | `anonymous` | `anonymous` (fresh) · `runAs: <userId>` (user context) |

When `user` is set and the engine lacks native session support, fall back to continue-style. When `runAs` is set, resolve the specified user's directory and session.

### Output and Notification

- `userIds` provided → send result to those Telegram users via DM.
- `userIds` omitted → silent (log-only).
- User crons default `userIds` to the owning user.

### Execution Logging

```
{projectCwd}/.hal/logs/crons/{job-name}/{timestamp}.{job-name}.txt
```

- One folder per job name. Timestamp-first filename for sorting; job name repeated for portability.
- Plain text. Respects `logging.persist` setting.
- Contents: job name, definition summary, start/end timestamps, full output, exit status.

### Cron Source Tiers

| Tier    | Inline config                     | File-based                            | Mutable from chat |
|---------|-----------------------------------|---------------------------------------|--------------------|
| System  | Top-level `crons` key             | `{configDir}/.hal/crons/*.yaml`       | No                 |
| Project | `projects[n].crons`               | `{projectCwd}/.hal/crons/*.yaml`      | No                 |
| User    | —                                 | `{dataDir}/{userId}/crons/*.yaml`     | Yes (owner only)   |

All tiers use the same YAML format. All file-based sources are hot-reloaded via file watchers.

### Lifecycle and Hot Reload

- **Startup:** load all tiers, validate, schedule.
- **Config hot-reload:** reconcile inline crons (add/update/remove). User crons unaffected.
- **File watchers:** detect file add/edit/delete across all tiers and reschedule.
- **One-off jobs:** marked `enabled: false` after execution, not rescheduled on reload.
- **Process restart:** all definitions re-read from disk.
