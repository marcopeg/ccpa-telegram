# 032 — Cron jobs and scheduled actions

## Status

draft

## Summary

Add cron-job support to schedule tasks in time, including one-off timers and recurring schedules, with persistent storage so jobs survive hot reload and process restarts. Jobs can be defined at three levels (system, project, user), execute prompts or shell commands, and log every execution to disk.

## Cron Source Tiers

Cron definitions are loaded from three tiers, mirroring the existing config hierarchy:

### System-level (global)

- **File-based:** `{configDir}/.hal/crons/*.yaml` — hot-reloaded.
- **Inline:** defined in the top-level config under a `crons` key (sibling to `globals`, `projects`).
- Managed by the operator; immutable from chat commands.

### Project-level

- **File-based:** `{projectCwd}/.hal/crons/*.yaml` — hot-reloaded.
- **Inline:** defined inside each `projects[n]` block under a `crons` key.
- Managed by the operator; immutable from chat commands.

### User-level

- **File-based:** `{dataDir}/{userId}/crons/*.yaml` — hot-reloaded.
- Created and managed via `/cron_*` slash commands (AI-assisted).
- Mutable by the owning user only.

All tiers use the same YAML file format (see below).

## Cron Definition Format

Each cron job is a single YAML file with the following shape:

```yaml
name: daily-git-status
enabled: true

# --- Scheduling (exactly one of these) ---
schedule: "0 9 * * *"          # cron expression (recurring)
# runAt: "2026-03-05T10:00:00Z" # ISO 8601 datetime (one-off)
# runIn: "30m"                   # relative duration (one-off)

# --- Action (exactly one of these) ---
action:
  type: prompt                  # "prompt" | "shell"
  value: "Check git status and summarize pending changes"

# --- Session behaviour ---
session: user      # see "Session Behaviour" section

# --- Output targets (optional) ---
userIds: [123456]  # Telegram user IDs to receive output; omit for silent
```

### Scheduling options

| Field      | Type     | Description                            |
|------------|----------|----------------------------------------|
| `schedule` | string   | Standard cron expression (recurring).  |
| `runAt`    | string   | ISO 8601 datetime (one-off, absolute). |
| `runIn`    | string   | Duration like `30m`, `2h` (one-off, relative to load time). |

Exactly one of the three must be present. One-off jobs (`runAt`/`runIn`) execute once and are then marked `enabled: false` in the file.

### Action types

| Type     | Description                                                              |
|----------|--------------------------------------------------------------------------|
| `prompt` | Sent to the current project engine as if the user typed a message.       |
| `shell`  | Executed as a subprocess in the project `cwd`.                           |

Skills can be invoked via a prompt action using the `/skill_name` syntax in the value.

## Session Behaviour

How the engine session is resolved when a cron fires depends on the cron tier:

### User-defined crons

| Value       | Behaviour                                                                                         |
|-------------|---------------------------------------------------------------------------------------------------|
| `user`      | **Default.** Use the owning user's existing session. Falls back to continue-style if the engine lacks native session support. |
| `anonymous` | Start a fresh session with no history.                                                            |

### Project-defined and system-defined crons

| Value                | Behaviour                                                                                          |
|----------------------|----------------------------------------------------------------------------------------------------|
| `anonymous`          | **Default.** Fresh session, no history.                                                            |
| `runAs: <userId>`    | Execute in the context of the specified user, using their session.                                 |

## Output and Notification

- If `userIds` is provided, the cron result is sent to those Telegram users via DM.
- If `userIds` is omitted, the execution is silent (log-only, no Telegram messages).
- Config/project crons define `userIds` explicitly; user crons default to the owning user.

## Slash Commands

A set of built-in commands for managing user-scoped crons:

| Command        | Description                                                                                   |
|----------------|-----------------------------------------------------------------------------------------------|
| `/cron_add`    | AI-assisted creation. Takes a natural-language prompt, passes it to the engine together with a skill/context prompt that explains the YAML format, and writes the resulting file to `{dataDir}/{userId}/crons/`. |
| `/cron_list`   | Programmatic listing. Shows all crons visible to the user: user-defined, project-defined, and system-defined, with status (enabled/disabled, next run). |
| `/cron_edit`   | AI-assisted edit. Takes a cron name and a natural-language change request, rewrites the YAML.  |
| `/cron_remove` | Deletes a user-defined cron by name. Cannot remove project or system crons.                   |
| `/cron_pause`  | Sets `enabled: false` on a user-defined cron.                                                 |
| `/cron_resume` | Sets `enabled: true` on a user-defined cron.                                                  |

### AI-assisted commands (`/cron_add`, `/cron_edit`)

These commands send the user's natural-language request to the engine along with a context/skill prompt that:
- Describes the cron YAML schema and valid field values.
- Provides the current cron definitions for context (on edit).
- Instructs the engine to output a valid YAML file.

The bot then writes/overwrites the resulting file to the user's crons directory.

## Execution Logging

Every cron execution is logged to disk regardless of whether output is sent to Telegram.

### Log location

```
{projectCwd}/.hal/logs/crons/{job-name}/{timestamp}.{job-name}.txt
```

- One folder per job name.
- Filename: `{timestamp}.{job-name}.txt` — timestamp-first for chronological sorting; job name repeated in filename so context is preserved if files are moved or shipped elsewhere.
- Plain text format, following the project's existing logging conventions.
- Respects the project's `logging.persist` setting (only written when persistence is enabled).

### Log contents

Each log file should capture:
- Job name and definition summary (schedule, action type).
- Execution start/end timestamps.
- Full action output (engine response or shell stdout/stderr).
- Exit status (success/failure, exit code for shell).

## Lifecycle and Hot Reload

- On startup: load cron definitions from all three tiers, validate, and schedule.
- On config change (hot reload): re-read inline crons from config, reconcile with running scheduler (add new, update changed, remove deleted config crons). User crons are unaffected by config reload.
- File watchers on `{configDir}/.hal/crons/`, `{projectCwd}/.hal/crons/`, and `{dataDir}/*/crons/` detect file-based cron changes and reschedule accordingly.
- One-off jobs (`runAt`/`runIn`) are marked `enabled: false` after execution and not rescheduled on reload.
- Process restart: all file-based and inline crons are re-read and rescheduled from disk state.

## Requirements

- Support three tiers of cron sources: system, project, and user.
- Support file-based (`.hal/crons/*.yaml`) and inline (config block) cron definitions at system and project levels.
- User crons are file-based only, stored under `{dataDir}/{userId}/crons/`.
- All cron files and directories are hot-reloaded via file watchers.
- Support both recurring (`schedule`) and one-off (`runAt`, `runIn`) scheduling.
- Support `prompt` and `shell` action types.
- Session behaviour is configurable per cron (anonymous, user, runAs).
- Config/system crons are immutable from chat commands.
- Provide `/cron_add`, `/cron_list`, `/cron_edit`, `/cron_remove`, `/cron_pause`, `/cron_resume` commands.
- `/cron_add` and `/cron_edit` use AI-assisted generation via engine + skill context.
- Log every cron execution to `{projectCwd}/.hal/logs/crons/` when `logging.persist` is enabled.
- Crons are project-scoped (each project has its own scheduler instance).

## Acceptance Criteria

- System, project, and user cron definitions all load and schedule correctly at boot.
- Inline config crons and file-based crons coexist without conflict.
- User-created crons (via `/cron_add`) are persisted to `{dataDir}/{userId}/crons/` and restored after restart.
- One-off jobs execute once, are marked `enabled: false`, and are not rescheduled.
- Recurring jobs execute on schedule and survive hot reload and restart.
- File watcher detects cron file additions, edits, and deletions across all tiers.
- `/cron_list` shows all visible crons with status and next-run information.
- `/cron_add` and `/cron_edit` correctly generate valid YAML via AI assistance.
- `/cron_remove` only deletes user-owned crons; rejects attempts on project/system crons.
- Execution logs are written per-run in the correct format and location.
- Silent crons (no `userIds`) produce log output but no Telegram messages.
- Session behaviour respects the configured mode (anonymous, user, runAs).
