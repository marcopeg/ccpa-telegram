## Summary

Add user-level cron support and the full set of `/cron_*` slash commands. Users can create, list, edit, pause, resume, and remove their own crons via AI-assisted chat commands. User crons are stored as YAML files in the user's data directory and hot-reloaded.

## Dependencies

- [032a — Core scheduler + system-level crons](./032a.core-scheduler-and-system-crons.md) must be completed first.
- [032b — Project-level crons + logging refinement](./032b.project-crons-and-logging.md) should be completed first (provides `runAs` session and multi-project scheduler patterns).

## Scope

### In scope

- User-level cron source:
  - **File-based:** `{dataDir}/{userId}/crons/*.yaml`, loaded at boot for known users.
  - Hot-reloaded via file watcher on `{dataDir}/*/crons/` directories.
  - Mutable only by the owning user through slash commands.
- `user` session mode for user-defined crons (as defined in 032).
- Slash commands (built-in, registered like other Hal commands):
  - `/cron_add` — AI-assisted creation.
  - `/cron_list` — programmatic listing of all visible crons (user + project + system).
  - `/cron_edit` — AI-assisted editing.
  - `/cron_remove` — delete a user-owned cron.
  - `/cron_pause` — set `enabled: false` on a user-owned cron.
  - `/cron_resume` — set `enabled: true` on a user-owned cron.
- AI-assisted cron skill/context prompt: a dedicated skill file that describes the YAML schema and guides the engine to produce valid cron definitions.
- User crons default `userIds` to the owning user (output goes back to them unless overridden).
- Add a `commands.cron` config option (with `enabled` flag) to control whether cron commands are registered.

### Out of scope

- Modifying project or system crons from chat (these remain immutable from slash commands).

## Slash Command Details

### `/cron_add <natural-language description>`

1. User sends something like: `/cron_add every morning at 9am check git status and tell me`.
2. Bot constructs a prompt combining the user's input + a skill/context prompt explaining the YAML format.
3. Engine generates a valid YAML cron definition.
4. Bot validates the output against the Zod schema.
5. Bot writes the file to `{dataDir}/{userId}/crons/{name}.yaml`.
6. File watcher picks up the new file and schedules it.
7. Bot confirms to the user with a summary of the created cron.

### `/cron_list`

Programmatic (no AI). Displays a formatted list of all crons visible to the user:
- **User crons** — owned by the requesting user.
- **Project crons** — defined for the current project.
- **System crons** — global definitions.

Each entry shows: name, tier (user/project/system), schedule/timing, action summary, enabled status, next scheduled run.

### `/cron_edit <cron-name> <change description>`

1. Bot loads the existing cron YAML by name (must be user-owned).
2. Constructs a prompt with the current definition + user's change request + skill context.
3. Engine produces the updated YAML.
4. Bot validates and overwrites the file.
5. File watcher detects the change and reschedules.
6. Bot confirms with a diff summary.

### `/cron_remove <cron-name>`

Deletes the file from `{dataDir}/{userId}/crons/`. Rejects if the cron is project-level or system-level.

### `/cron_pause <cron-name>` / `/cron_resume <cron-name>`

Reads the user's cron file, sets `enabled: false` or `true`, writes back. File watcher handles reschedule.

## Requirements

- Implement user cron loading: on boot, scan `{dataDir}/*/crons/` for existing user cron files.
- Add file watcher on `{dataDir}/*/crons/` to detect user cron changes.
- Implement `user` session mode: resolve the owning user's session and execute with their context.
- Register all six `/cron_*` commands as built-in commands (enabled via config, similar to git commands).
- Create the AI-assistance skill/context prompt file for cron generation.
- `/cron_add` and `/cron_edit` must validate AI output against the Zod schema before writing.
- `/cron_list` must aggregate crons from all three tiers with clear tier labeling.
- `/cron_remove`, `/cron_pause`, `/cron_resume` must enforce user-ownership (reject non-user crons).

## Acceptance Criteria

- `/cron_add` creates a valid YAML file in the user's crons directory via AI assistance.
- The created cron is automatically picked up by the file watcher and scheduled.
- `/cron_list` shows user, project, and system crons with status and next-run info.
- `/cron_edit` modifies an existing user cron via AI and the scheduler updates.
- `/cron_remove` deletes a user cron file and the scheduler removes it.
- `/cron_pause` and `/cron_resume` toggle the `enabled` field and the scheduler reacts.
- Attempting to remove/edit/pause a project or system cron via slash command is rejected with a clear message.
- User crons execute with the user's session by default.
- User crons survive process restart (reloaded from files on boot).
- Invalid AI-generated YAML is caught by validation and the user is informed.
- The cron commands appear in `/help` and the Telegram command menu when enabled.

## Notes

- The quality of AI-generated crons depends heavily on the skill/context prompt — it should include examples and edge cases.
- Consider adding a confirmation step in `/cron_add` where the bot shows the generated definition before writing it.
