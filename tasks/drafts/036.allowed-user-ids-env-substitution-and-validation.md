# 036 — Allowed user IDs: env substitution and numeric validation

## Status

draft

## Summary

Expand `allowedUserIds` config handling to support environment-variable substitution by allowing string input, while enforcing that effective values are valid Telegram numeric user IDs.

## Requirements

- Allow `allowedUserIds` entries to be provided as strings (not only numbers), so env-var substitution can be used.
- After substitution/resolution, parse each value to a number.
- Enforce strict numeric integrity:
  - if parsed numeric content does not exactly match the original resolved string representation, treat it as invalid.
  - invalid entries must produce a clear configuration error.
- Preserve compatibility for existing numeric `allowedUserIds` values.
- Keep runtime access checks numeric after parsing/validation.

## Validation Expectations

- Accept values like `"123456789"` and `123456789`.
- Reject values with non-numeric content or formatting drift (e.g. spaces, signs, decimals, scientific notation, mixed text).
- Error messaging should clearly identify which value/key failed validation.

## Acceptance Criteria

- Config schema accepts string-or-number forms for `allowedUserIds`.
- Env-substituted values are validated and normalized to numeric IDs.
- Invalid values fail fast at config load with actionable errors.
- Existing setups with numeric IDs continue to work without changes.
