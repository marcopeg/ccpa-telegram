# 031 — Reset model with duration and cron scheduling

## Status

draft

## Summary

After a user changes/selects a model, add a second inline-keyboard step asking how long to keep that model override, then automatically reset when the selected duration expires.

## Requirements

- After `/model` selection, show a follow-up keyboard with duration options.
- Suggested options: `5 minutes`, `30 minutes`, `1 hour`, `6 hours`, `until tomorrow`, `forever`.
- If the user selects a finite duration, persist reset metadata and schedule a cron-based reset back to the default/previous model.
- If the user selects `forever`, keep current behavior (no scheduled reset).
- Reset scheduling must survive process restarts (persisted schedule state + boot-time rehydration logic).

## Dependency / Gating

- This task is fully dependent on existing cron-job support in the project.
- Task refinement must verify cron support first by checking backlog/docs (including README).
- If cron support is not available, refinement must be blocked and explicitly report this dependency as unresolved.

## Acceptance Criteria

- Model change flow includes the new duration keyboard step.
- Finite-duration choices create scheduled reset jobs.
- Scheduled reset restores the intended model and cleans up schedule state.
- `Forever` choice does not create a reset job.
- Refinement process enforces the cron-support precondition.
