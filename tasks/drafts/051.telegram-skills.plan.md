## Phase 3 ŌĆö Pre-LLM interaction in text handler

The interaction state is per-user, in-memory. It lives in `src/bot/interaction/`.

### Step 3.1 ŌĆö Create `src/bot/interaction/state.ts`

```ts
export type PendingKind = "input" | "buttons";

export interface PendingInteraction {
  kind: PendingKind;
  skillEntry: CommandEntry;
  args: string[];
  timeoutHandle: ReturnType<typeof setTimeout>;
  resolve: (value: string | null) => void;
}

const pending = new Map<number, PendingInteraction>(); // keyed by userId

export function setPending(userId: number, p: PendingInteraction): void
export function getPending(userId: number): PendingInteraction | undefined
export function clearPending(userId: number): void
```

### Step 3.2 ŌĆö Update text handler (`text.ts`) ŌĆö skill dispatch

In the skill resolution section, after resolving `skillEntry`:

**Before dispatching to LLM:**

1. If user has a pending interaction for a different slash command ŌåÆ cancel it (call `resolve(null)`, clear pending)
2. If `skillEntry.telegram?.buttons` is set and no args ŌåÆ render inline keyboard, store pending, await resolve
3. Else if `skillEntry.telegram?.input === true` and no args ŌåÆ send follow-up prompt, store pending, await free-text reply
4. Otherwise ŌåÆ proceed directly to `agent.call()`

**When awaiting buttons:**
- Send message with `InlineKeyboard` (grammy) listing buttons
- Each button has label as display text and a unique `callback_data` (e.g. `skill:{commandName}:{buttonIndex}`)
- Register a one-shot callback query handler that resolves with the selected `prompt`
- Set timeout: call `resolve(null)` after `telegram.timeout ?? 60` seconds

**When awaiting text input:**
- Send a "Please reply with ŌĆ”" prompt
- Set a flag so the next non-command message from this user is treated as the skill reply
- Set timeout: call `resolve(null)` after `telegram.timeout ?? 60` seconds

**After resolution:**
- If resolved with `null` ŌåÆ send "Cancelled (timeout or new command)" message and return
- If resolved with a prompt string ŌåÆ build final prompt: `${skillEntry.skillPrompt}\n\nUser input: ${resolvedInput}` and proceed to `agent.call()`

**Cancel path:**
- When a slash command is received and there is a pending interaction for the user ŌåÆ cancel it before processing the new command.
- Send "Previous skill cancelled." message to the user.

### Step 3.3 ŌĆö Handle pending text-reply interception

In the text handler, before the normal slash-command interception block, check if the user has a pending `"input"` interaction. If so, treat the incoming (non-command) message as the reply, resolve the pending interaction, and return.

## Notes

- `yaml` package (`^2.8.2`) is already in `dependencies` ŌĆö use it for frontmatter parsing
- Grammy's `InlineKeyboard` class handles button rendering; callback queries are handled via `bot.on("callback_query:data", ...)`
- Interaction state is in-memory only; a bot restart clears all pending interactions (acceptable for MVP)
- If `telegram.buttons` and `telegram.input` are both set, buttons take precedence (per task spec)
