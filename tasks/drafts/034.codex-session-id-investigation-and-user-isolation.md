# 034 — Codex session ID investigation and user isolation

## Status

draft

## Summary

Investigate how to obtain and track Codex session IDs (even if unofficial) to prevent session sharing across users and improve per-user conversation isolation for non-Claude engines.

## Problem

Currently, for tools that are not Claude, session context can be shared across users. This can leak conversational state between users and cause incorrect responses.

## Goals

- Investigate whether Codex exposes session identifiers through CLI output, logs, metadata, or local state files.
- Explore filesystem-based discovery (for example in user home folders) when a new Codex session starts.
- Determine a reliable strategy to map each Telegram user to a distinct Codex session.
- Define fallback behavior when session ID extraction is unavailable.

## Scope

- Research-only + technical design proposal.
- Include risk analysis for relying on unofficial/internal Codex artifacts.
- Propose implementation options with trade-offs: reliability, portability, maintenance cost.
- Consider implications for other non-Claude engines where session isolation is also needed.

## Deliverables

- Findings report on viable methods to detect/get Codex session ID.
- Recommended per-user session isolation strategy.
- Migration plan from shared session behavior to isolated sessions.
- Test/validation checklist for multi-user scenarios.

## Acceptance Criteria

- At least one practical session-ID discovery approach is documented and evaluated.
- Session-isolation design clearly prevents cross-user context leakage.
- Risks of unofficial integration points are explicitly documented.
