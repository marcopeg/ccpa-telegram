# 032a — Core scheduler + system-level crons

## Status

draft

> **Part of [032 — Cron jobs and scheduled actions](./032.cron-jobs-and-scheduled-actions.md).**
> Read the parent task first for shared specifications (YAML format, session behaviour, logging format, source tiers, lifecycle rules). This file only covers what is specific to 032a.

## Summary

Introduce the core cron scheduler infrastructure and system-level cron support. Builds the scheduler engine, YAML format parser, action executors, execution logging, and system-tier cron loading (inline config + file-based). No project-level or user-level support yet.

## Scope

### In scope

- Add a scheduling library dependency (e.g., `croner`).
- Cron YAML format: Zod schema for parsing and validating cron definition files (as specified in 032).
- Core scheduler engine: load definitions, register timers, execute on schedule, manage lifecycle (start, stop, reschedule).
- Scheduling modes: recurring (`schedule`), one-off absolute (`runAt`), one-off relative (`runIn`).
- Action executors:
  - `prompt` — send to the project engine via `agent.call()`, anonymous session only.
  - `shell` — run as subprocess in project `cwd`, capture stdout/stderr.
- System-level cron sources:
  - **Inline:** new top-level `crons` key in config (sibling to `globals`, `projects`), validated by Zod.
  - **File-based:** `{configDir}/.hal/crons/*.yaml`, loaded at boot.
- File watcher on `{configDir}/.hal/crons/` for hot-reload of system cron files.
- Config hot-reload: reconcile inline crons when config changes (add new, update changed, remove deleted).
- Execution logging to disk (format defined in 032).
- One-off job lifecycle: mark `enabled: false` in the YAML file after execution.
- `userIds` output targeting: if provided, send result to those Telegram user IDs; otherwise silent (log-only).

### Out of scope (deferred to 032b/032c)

- Project-level crons (inline in `projects[n]` or `{projectCwd}/.hal/crons/`).
- User-level crons (`{dataDir}/{userId}/crons/`).
- Slash commands (`/cron_add`, `/cron_list`, etc.).
- Session modes beyond `anonymous` (`user`, `runAs`).

## Requirements

- Add scheduling library as a dependency.
- Implement a `CronScheduler` class/module that:
  - Accepts an array of validated cron definitions.
  - Registers timers for each enabled definition.
  - Executes the appropriate action handler when a timer fires.
  - Exposes methods to add, remove, and update individual jobs at runtime.
  - Provides a `stop()` / `destroy()` method for clean shutdown.
- Implement a Zod schema for the cron definition format.
- Implement a YAML loader that reads `*.yaml` files from a directory and returns validated definitions.
- Extend the config schema to accept a top-level `crons` array.
- Wire the scheduler into the bot startup lifecycle: create scheduler after config is loaded, stop on shutdown.
- Wire into config hot-reload: on config change, reconcile inline system crons with the running scheduler.
- Add a file watcher on `{configDir}/.hal/crons/` that triggers reschedule on file add/change/delete.
- Implement execution logging to disk.
- Implement `userIds` Telegram output (send result as DM to each listed user).

## Acceptance Criteria

- A cron defined inline in the top-level `crons` config key fires on schedule.
- A cron defined as a YAML file in `{configDir}/.hal/crons/` fires on schedule.
- Both `prompt` and `shell` action types execute correctly.
- Recurring crons keep firing on schedule.
- One-off crons (`runAt`, `runIn`) fire once and are marked `enabled: false`.
- Adding/editing/deleting a YAML file in the system crons directory is detected and the scheduler updates without restart.
- Config hot-reload picks up changes to inline `crons` and reconciles.
- Execution logs are written to the correct path with the expected content.
- When `userIds` is set, results are sent to those Telegram users; when omitted, execution is silent.
- Clean shutdown stops all scheduled timers.
- The scheduler handles invalid/malformed YAML gracefully (logs warning, skips).

## Notes

- Keep the `CronScheduler` generic enough that 032b and 032c can reuse it with project-scoped and user-scoped definitions.
- The logging format established here will be refined in 032b based on real usage.
