# 051 — Telegram skill frontmatter properties

## Status

draft

## Summary

Extend skill frontmatter with a richer nested `telegram` object. Replace the current top-level `telegram: true/false` boolean with structured Telegram-specific properties that control visibility and pre-LLM interaction behavior.

## Context

Task 038 introduced the `telegram: true` flag as the sole criterion for exposing a skill to Telegram (both `/help` and the slash command menu). This single flag is too coarse: there is no way to distinguish between a skill that should appear in the menu but not `/help`, one that needs user input before running, or one that should only be listed in help for discoverability. A richer frontmatter vocabulary is needed.

## Refined schema (agreed)

Use a nested `telegram` object in SKILL.md frontmatter.

| Property | Type | Description |
|---|---|---|
| `telegram.enabled` | boolean | Master visibility switch for Telegram surfaces. Optional; defaults to `true` when `telegram` object is present. |
| `telegram.help` | boolean | Controls inclusion in `/help` (`HAL_COMMANDS` Project Skills). Optional; defaults to `true` when omitted. |
| `telegram.menu` | boolean | Controls inclusion in Telegram slash command menu (`setMyCommands`). Optional; defaults to `true` when omitted. |
| `telegram.input` | boolean | When `true`, skill requires follow-up free-text input before LLM dispatch when initial invocation has no args. |
| `telegram.buttons` | `{ label: string, prompt: string }[]` | Optional pre-LLM button choices. Each button has UI label + prompt payload to inject into the final LLM input. Takes precedence over `telegram.input` when both are present. |
| `telegram.timeout` | number | Optional timeout for waiting on button callback or follow-up input, in seconds. Defaults to `60` when omitted. |

Canonical behavior:

- If `telegram` object is absent → skill is hidden from Telegram surfaces by default.
- If `telegram` object is present and no visibility subflags are provided → visible in both help and menu by default.
- `telegram.help` and `telegram.menu` are primarily used to opt out of one surface.
- If `telegram.enabled: false`, skill is hidden from all Telegram surfaces regardless of `help/menu`.
- Legacy top-level `telegram: true/false` is invalid and must trigger a hard failure (startup/load error), not aliasing.

### Skill invocation flow with pre-LLM user interaction

When HAL resolves a skill to invoke, it must parse the frontmatter **before** dispatching to the LLM to determine whether user interaction is required first:

1. If `telegram.buttons` is present → render an inline keyboard with the listed options, await callback query, then inject selected button `prompt` into skill context and forward to LLM.
2. Else if `telegram.input: true` and invocation has no args → send follow-up prompt, await free-text reply, then forward it with skill context.
3. Otherwise → pass the skill directly to the LLM without any intermediate interaction.

MVP interaction constraints (agreed):

- Single-step interaction only (no multi-step conversation flows in this task).
- Include explicit cancel path for pending input/button interaction.
- Apply timeout for pending input/button waits; default `60s`, overridable via `telegram.timeout`.
- If user sends another slash command while waiting, pending interaction is cancelled and normal command handling continues.

This means the bot layer needs awareness of the skill's frontmatter at dispatch time, not just at registration time.

> **Refinement scope note:** This task intentionally targets MVP primitives only (inline keyboard callback + single follow-up reply capture). Multi-step conversations and dynamic runtime button generation are out of scope.

## Migration

- Replace all legacy top-level `telegram: true/false` skill frontmatter with nested `telegram:` object schema.
- Affected files: all SKILL.md files in `.agents/skills/`, `examples/`, and any other locations that currently use `telegram`.
- Update all docs that describe the `telegram` frontmatter key.
- Enforce strict validation: any remaining top-level `telegram: true/false` in loaded skills causes hard failure with actionable error.

## Requirements

- Parse nested `telegram` schema from SKILL frontmatter (`enabled`, `help`, `menu`, `input`, `buttons`, `timeout`) with defaults defined above.
- Update the skill loader and `CommandEntry` type to carry resolved Telegram metadata (not a single boolean flag).
- Reject legacy top-level `telegram: true/false` with hard failure (no alias, no deprecation period).
- Implement `telegram.buttons` flow: at dispatch time, render inline keyboard, await callback query, inject selected button `prompt` into skill context, then proceed.
- Implement `telegram.input` flow: on invocation without args, send a follow-up prompt, await reply, then proceed with skill execution.
- For both interaction flows: support cancel + timeout; default timeout 60s, configurable by `telegram.timeout`.
- Update `/help` filtering to respect `telegram.help` (subject to `telegram.enabled`).
- Update `setMyCommands` filtering to respect `telegram.menu` (subject to `telegram.enabled`).
- Migrate all existing skill frontmatter from legacy `telegram: bool` to nested `telegram` object.
- Update all relevant docs under `docs/` and example configs.

## Open questions (for refinement)

None at this stage; key schema and behavior decisions are now specified.

## Acceptance Criteria

- Skill frontmatter supports the agreed nested `telegram` schema; legacy top-level `telegram: true/false` is rejected with hard failure.
- All existing skills are migrated; no remaining legacy `telegram: true/false` in loaded skills.
- `/help` respects `telegram.help`; menu respects `telegram.menu`; both are gated by `telegram.enabled`.
- `telegram.buttons` skills present an inline keyboard before LLM invocation; selected button `prompt` is injected into skill context.
- `telegram.input: true` skills prompt for free-text input when invoked without args and await the reply before LLM invocation.
- Interaction waits support cancel and timeout; timeout defaults to 60s and is overridable with `telegram.timeout`.
- Docs and examples updated.
