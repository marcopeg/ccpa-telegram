# 070 — Debounce voice messages

**Status**: draft

## Description

When a voice recording exceeds Telegram's ~60-second limit, Telegram automatically splits it into multiple consecutive voice messages. Currently each part is processed independently by the voice handler — the engine receives and responds to each audio fragment separately. HAL should buffer consecutive voice messages from the same user, collect all parts within a debounce window, and send the full stitched transcript to the engine as one request.

This task also restructures the existing `debounce` config shape from a flat object to a nested one, grouping settings by message type (`text`, `voice`). No backward compatibility is required.

## Context & Constraints

- **Scope**: voice messages only (`message:voice` handler). Related but distinct from task 068 (which stitches Whisper partial transcript output within a single voice message).
- **Trigger**: Telegram splits long voice recordings at ~60-second boundaries, delivering each chunk as a separate voice message update in rapid succession.
- **Bot library**: Grammy v1.39.2.

## Approach

### Audio stitching: transcribe-then-join

Each audio chunk is transcribed independently as it arrives (same flow as the current voice handler); transcripts are concatenated in `message_id` order and sent to the engine as a single prompt once the debounce window expires. This mirrors how text parts are joined in `text.ts` and is consistent with task 068's transcript-level stitching.

### Debounce activation heuristic

Debouncing is activated **only when the first received voice message has `voice.duration >= 60`** (seconds). This mirrors the text handler's `messageText.length >= 4096` threshold — it signals that Telegram has split the recording and more parts are coming.

- A short voice message (`duration < 60`) is dispatched immediately, with no buffering overhead.
- Once buffering is active for a user, any subsequent voice message resets the timer (same as text).

### Part ordering

Parts are sorted by `message_id` (ascending) before concatenating transcripts — the same strategy used in the text debounce `flush()`. This ensures correct ordering regardless of upload timing variations.

### No status message during buffering

While collecting parts, no status message is shown. A single status message appears only when the buffer flushes and actual processing begins.

## Config schema restructure

The `debounce` key is restructured from a flat object to a nested one. No backward compatibility needed — old `debounce.windowMs` is simply replaced.

### Before
```ts
// ResolvedConfig type
debounce: { windowMs: number }

// User config (yaml)
debounce:
  windowMs: 1500
```

### After
```ts
// ResolvedConfig type
debounce: {
  text:  { windowMs: number };
  voice: { windowMs: number };
}

// User config (yaml)
debounce:
  text:
    windowMs: 1500
  voice:
    windowMs: 4000
```

### Zod schema (globals + project, same shape)
```ts
debounce: z
  .object({
    text:  z.object({ windowMs: z.number().positive() }).partial().optional(),
    voice: z.object({ windowMs: z.number().positive() }).partial().optional(),
  })
  .partial()
  .optional()
```

### Default resolution (project → globals → hardcoded)
```ts
debounce: {
  text:  { windowMs: project.debounce?.text?.windowMs  ?? globals.debounce?.text?.windowMs  ?? 300  },
  voice: { windowMs: project.debounce?.voice?.windowMs ?? globals.debounce?.voice?.windowMs ?? 4000 },
}
```

Default of **4000ms** for voice accounts for the time Telegram needs to upload a ~60-second audio clip on a mobile connection before the next chunk arrives.

### `src/bot/handlers/voice.ts`
New debounce logic, modelled on `text.ts`:
- Add a per-user buffer (`Map<userId, BufferEntry>`) where `BufferEntry` holds: `parts: Array<{ transcript: string; messageId: number }>`, `timer`, `gramCtx`.
- On each incoming voice message:
  - If `voice.duration >= 60` OR a buffer already exists for this user → buffer the part (download + transcribe immediately, store transcript + `message_id`, reset timer).
  - Otherwise → dispatch immediately (existing flow, unchanged).
- On flush: sort parts by `message_id`, concatenate transcripts, call the existing dispatch logic with the combined text.
- Read window from `config.debounce.voice.windowMs`.
- Pass `debounceActiveUsers` (same `Set<number>` used by text) to allow the rate-limit middleware to skip counting buffered parts. Wire this in `src/bot.ts` — the voice handler registration needs the same `debounceActiveUsers` set passed in.

## Acceptance Criteria

- [ ] `debounce.windowMs` flat key is gone; replaced by `debounce.text.windowMs` and `debounce.voice.windowMs` throughout (schema, type, resolver, handlers).
- [ ] Debouncing for voice activates only when the first voice message has `duration >= 60`; shorter messages are dispatched immediately.
- [ ] Multiple consecutive voice messages (Telegram-split recording) are buffered per user and flushed once the debounce window expires with no new voice message.
- [ ] Each part is transcribed individually as it arrives; transcripts are concatenated in `message_id` order and sent as one prompt to the engine.
- [ ] Voice debounce window is configurable via `debounce.voice.windowMs`; default is 4000ms.
- [ ] Text debounce window is configurable via `debounce.text.windowMs`; default is 300ms (unchanged behaviour).
- [ ] Buffer is per-user; simultaneous voice messages from different users do not interfere.
- [ ] No status message is shown while buffering voice parts; status appears only on flush.
- [ ] A single voice message with `duration < 60` is processed correctly with no buffering.
- [ ] Buffered voice parts are exempt from rate-limit counting (same `debounceActiveUsers` set, passed to voice handler and rate-limit middleware).
- [ ] Parts are sorted by `message_id` before concatenation to guarantee correct ordering.
- [ ] Existing voice handler behaviour (transcription, streaming status, engine execution) is unchanged for the flushed combined prompt.
- [ ] `docs/config/debounce/README.md` created; `docs/config/README.md` table and `docs/config/reference.yaml` updated.
