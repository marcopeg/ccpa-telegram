# Plan — 067: Rename config-dir flag to config

## Phase 1 — Core CLI changes (`src/cli.ts`)

### Step 1.1 — Update `ParsedArgs` interface
Add `configFile?: string` alongside the existing `configDir: string`.

### Step 1.2 — Rewrite the `--config` parsing block in `parseArgs()`
Replace the two `--config-dir` / `--config-dir=` branches with `--config` / `--config=` branches.

Detection logic (applied after resolving the raw value to an absolute path):
- Value has **no extension** → directory mode: set `configDir`, leave `configFile` undefined.
- Value has an extension in `CONFIG_EXTENSIONS` (`.json`, `.jsonc`, `.yaml`, `.yml`) → file mode:
  - Crash with a clear message if the file does not exist.
  - Set `configDir` to `dirname(resolved)`, set `configFile` to `basename(resolved)`.
- Value has **any other extension** → crash immediately with: `Error: unsupported config extension "<ext>". Supported: .json, .jsonc, .yaml, .yml`.

The set of valid extensions must be imported/derived from `config.ts` (or duplicated from `CONFIG_EXTENSIONS`) to stay in sync.

### Step 1.3 — Update `showHelp()`
- Replace `--config-dir <path>` in the Options section with:
  `--config <path>  Config directory, or explicit config file path (see supported extensions)`
- Replace both `--config-dir` occurrences in the Examples section with `--config`.

### Step 1.4 — Thread `configFile` through the call chain
- `parseArgs()` returns `configFile` in `ParsedArgs`.
- `main()` destructures `configFile` and passes it to `runStart()`.
- `runStart(configDir, wizardPrefill)` gains a third param `configFile?: string` and forwards it to `tryLoadMultiConfig(configDir, configFile)`.
- `runInit` uses `configDir` as before (init always writes a new file; file-path mode doesn't apply).

## Phase 2 — Config loader (`src/config.ts`)

### Step 2.1 — Update `tryLoadMultiConfig` signature
```ts
export function tryLoadMultiConfig(
  configDir: string,
  configFile?: string,
): LoadedConfigResult
```
Forward `configFile` to `loadMultiConfigInternal`.

### Step 2.2 — Update `loadMultiConfigInternal`
When `configFile` is provided:
- Resolve the full path as `join(configDir, configFile)`.
- Determine its format from the extension (using `CONFIG_EXTENSIONS`).
- Load that file directly, bypassing the `resolveConfigFile` scan.
- Produce the same `LoadedConfigResult` shape as the directory case.

When `configFile` is absent, behaviour is unchanged.
