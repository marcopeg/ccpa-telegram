## Log

### 2026-03-02 17:18 — Boot logs: human-readable format and stable order

- **Startup logger**: Replaced raw Pino JSON with a human-readable stream that parses NDJSON and prints `[HH:mm:ss] LEVEL  message` (and optional key=value) so boot lines are readable in the terminal.
- **Project loggers (flow)**: When `logging.flow` is true, terminal output now uses the same pretty stream instead of raw JSON, so "Starting bot", "Engine CLI verified", "Bot is running", "Commands registered" and any runtime logs are readable.
- **Order**: Startup messages ("Loading configuration", "Configuration sourced", "Configuration loaded") now appear first; no change to code order—the pretty stream writes immediately so output order is stable.
- **Persistence**: When `logging.persist` is true, log files still receive raw JSON (unchanged) for machine parsing.
- **Files**: `src/logger.ts` — added `LEVEL_NAMES`, `createPrettyLogStream(out?)`, use it in `createStartupLogger()` and in `createProjectLogger()` for the flow stream.

### 2026-03-02 17:22 — setMyCommands error: name + path for long descriptions

- **Problem**: When a skill (or any command) had a description longer than Telegram’s 256-character limit, the log only showed the generic API error ("command description length must not exceed 256") with no indication of which command or file.
- **Change**: Validate descriptions before calling `setMyCommands`. Added `TELEGRAM_MAX_DESCRIPTION_LENGTH` and `getCommandsWithDescriptionTooLong(commands, configDir)` in `src/bot/commands/loader.ts`. Path is relative to the project config directory; built-ins show `(builtin)`.
- **Startup**: In `src/bot.ts`, if any command is too long we throw a clear error listing each offender with `/command`, description length, and relative path (e.g. `.agents/skills/skillname/SKILL.md`).
- **Watcher**: Same validation in `src/bot/commands/watcher.ts` before re-registering on file change; logs offenders and skips the API call instead of failing with the 400.
- **Skill filePath**: Skills now store the path to their `SKILL.md` (no longer `""`) so the offending file can be reported; same in `resolveSkillEntry`.
